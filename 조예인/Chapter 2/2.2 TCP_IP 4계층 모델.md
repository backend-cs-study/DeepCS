## OSI 계층
✔️ 설명 <br>
- 애플리케이션 계층을 세 개로 쪼개고
- 링크 계층을 데이터 링크 계층, 물리 계층으로 나눠서 표현하는 것이 다름
- 인터넷 계층을 네트워크 계층으로 부른다는 점이 다름

- ex) 전송 계층에서 TCP -> UDP로 변경했다고 해서
- 인터넷 웹 브라우저를 다시 설치해야 하는 것은 아니듯
- 유연하게 설계됨

### 애플리케이션(application) 계층
- FTP, HTTP, SSH, SMTP, DNS 등 응용 프로그램이 사용되는 프로토콜 계층
- 웹 서비스, 이메일 등 서비스를 실질적으로 사람들에게 제공하는 층

✔️ 용어 정리 <br>
- FTP : 장치와 장치 간의 파일을 전송하는 데 사용되는 표준 통신 프로토콜
- SSH : 보안되지 않은 네트워크에서 네트워크 서비스를 안전하게 운영하기 위한 암호화 네트워크 프로토콜
- HTTP : World Wide Web을 위한 데이터 통신의 기초이자 웹 사이트를 이용하는 데 쓰는 프로토콜
- SMTP : 전자 메일 전송을 위한 인터넷 표준 통신 프로토콜
- DNS : 도메인 이름과 IP 주소를 매핑해주는 서버

### 전송(transport) 계층
- 송신자와 수신자를 연결하는 통신 서비스를 제공
- 연결 지향 데이터 스트림 지원, 신뢰성, 흐름 제어 제공 가능
- 애플리케이션과 인터넷 계층 사이의 데이터가 전달될 때 중계 역할
- TCP, UDP

#### TCP
- 패킷 사이의 순서를 보장 
- 연결지향 프로토콜을 사용해서 연결
- 신뢰성을 구축해서 수신 여부를 확인
- '가상회선 패킷 교환 방식' 사용

#### UDP
- 순서를 보장하지 않음
- 수신 여부를 확인하지 않음 
- 단순히 데이터만 줌
- '데이터그램 패킷 교환 방식' 사용

#### 가상회선 패킷 교환 방식
- 각 패킷에는 가상회선 식별자가 포함됨
- 모든 패킷을 전송하면 가상회선이 해제되고 
- 패킷들을 전송된 '순서대로' 도착하는 방식

#### 데이터그램 패킷 교환 방식
- 패킷이 독립적으로 이동
- 최적의 경로를 선택하여 감
- 하나의 메시지에서 분할된 여러 패킷은 서로 다른 경로로 전송 가능
- 도착한 '순서가 다를 수' 있는 방식

#### TCP 연결 성립 과정
- TCP는 신뢰성을 확보할 때 '3-웨이-핸드셰이크(3-way handshake)' 작업 진행
- 3-웨이-핸드셰이크 과정 이후 신뢰성이 구축되고 데이터 전송 시작
- TCP는 이 과정이 있기 때문에 신뢰성이 있는 계층이라고 함
- UDP는 이 과정이 없기 때문에 신뢰서잉 없는 계층이라고 함

##### 3-웨이-핸드셰이크(3-way handshake)
✔️ SYN 단계 <br>
- 클라이언트 -> 서버
- 클라이언트의 ISN을 답아 SYN을 보냄
- ISN : 새로운 TCP 연결의 첫 번째 패킷에 할당된 임의의 시퀀스 번호, 이는 장치마다 다를 수 있음

✔️ SYN + ACK 단계 <br>
- 서버 -> 클라이언트
- 클라이언트의 SYN를 수신
- 서버의 ISN을 보냄 
- 승인번호로 클라이언트의 ISN + 1을 보냄

✔️ ACK 단계 <br>
- 클라이언트 -> 서버
- 서버의 ISN + 1한 값인 승인번호를 담아 
- ACK를 서버에 보냄

✔️ 용어 정리 <br>
- SYN : SYNchronization의 약자, 연결 요청 플래그
- ACK : ACKnowledgement의 약자, 응답 플래그
- ISN : Initial Sequence Numbers의 약어, 초기 네트워크 연결을 할 때 할당된 32비트 고유 시퀀스 번호

#### TCP 연결 해제 과정
- TCP가 연결을 해제할 때 '4-웨이-핸드셰이크(4-way handshake)' 과정 발생
- 
✔️ 1번 <br>
- 클라이언트 -> 서버
- 클라이언트가 연결을 닫으려고 할 때
- FIN으로 설정된 세그먼트를 보냄
- 클라이언트는 FIN_WAIT_1 상태로 들어가고 서버의 응답 기다림

✔️ 2번 <br>
- 서버 -> 클라이언트
- 서버는 클라이언트로 ACK라는 승인 세그먼트 보냄
- CLOSE_WAIT 상태에 들어감
- 클라이언트가 세그먼트를 받으면 FIN_WAIT_2 상태에 들어감

✔️ 3번 <br>
- 서버 -> 클라이언트
- 서버는 ACK를 보내고 일정 시간 이후에 클라이언트에 FIN이라는 세그먼트 보냄

✔️ 4번 <br>
- 클라이언트 -> 서버
- 클라이언트는 TIME_WAIT 상태가 되고 나서
- 다시 서버로 ACK를 보내서
- 서버는 CLOSED 상태가 됨
- 이후 클라이언트는 어느 정도의 시간을 대기한 후 연결이 닫히고
- 클라이언트와 서버의 모든 자원의 연결이 해제됨

✔️ TIME_WAIT : 일정 시간 뒤에 닫는 이유 <br>
- 첫 번째, 지연 패킷이 발생할 경우를 대비하기 위함. 패킷이 뒤늦게 도달하고 이를 처리하지 못한다면 데이터 무결성 문제 발생
- ex) 전체 데이터가 100일 때 일부 데이터인 50만 들어오는 현상 발생 가능
- 두 번째, 두 장치가 연결이 닫혔는지 확인하기 위함.
- 만약 LAST_ACK 상태에서 닫히게 되면 다시 새로운 연결을 하려고 할 때 
- 장치는 줄곧 LAST_ACK로 되어 있기 때문에 접속오류가 나타나게 될 것

✔️ 용어 정리 <br>

- TIME_WAIT : 소켓이 바로 소멸되지 않고 일정 시간 유지되는 상태
- 지연 패킷 등의 문제점을 해결하는 데 사용
- CentOS6, 우분투에는 60초로 설정됨 윈도우는 4분 설정, 즉, OS마다 조금씩 다를 수 있음
- 데이터 무결성(data integrity) : 데이터의 정확성과 일관성을 유지하고 보증하는 것

### 인터넷(internet) 계층
- 장치로부터 받은 네트워크 패킷을 IP 주소로 지정된 목적지로 전송하기 위해 사용되는 계층
- IP, ARP, ICMP 등
- 패킷을 수신해야 할 상대의 주소를 지정하여 데이터 전달
- 상대방이 제대로 받았는지에 대해 보장하지 않는 비연결형적인 특징 가짐

---

### 링크 계층
- 전선, 광섬유, 무선 등으로 실질적으로 데이터를 전달하며 장치 간의 신호를 주고받는 '규칙'을 정하는 계층
- 네트워크 접근 계층이라고도 함

### 유선 LAN(IEEE802.3)
- 유선 LAN을 이루는 이더넷은 IEEE802.3이라는 프로토콜을 따르며 전이중화 통신 사용

#### 전이중화(full duplex) 통신
- 양쪽 장치가 동시에 송수신할 수 있는 방식
- 송신로와 수신로로 나눠서 데이터를 주고받으며
- 현대의 고속 이더넷을 이 방식을 기반으로 통신함

#### CSMA/CD(Carrier Sense Multiple Access with Collision Detection)
- 이전에는 유선 LAN에 '반이중화 통신' 중 하나인 CSMA/CD 방식 사용
- 데이터를 '보낸 이후' 충돌이 발생한다면 일정 시간 이후 재전송하는 방식
- 수신로와 송신로를 각각 둔 것이 아니고 한 경로를 기반으로 데이터를 보내기 때문에 
- 데이터를 보낼 때 충돌에 대해 대비해야 했기 때문

### 유선 LAN을 이루는 케이블
- TP 케이블(트위스트 페어 케이블), 광섬유 케이블

#### 트위스트 페어 케이블(twisted pair cable)
- 하나의 케이블처럼 보이지만 실제로는 여뎗 개의 구리선을 두 개씩 꼬아서 묶은 케이블
- UTP 케이블 : 케이블은 구리선을 실드 처리하지 않고 덮음,, 우리가 많이 볼 수 있는 케이블, LAN 케이블
- STP 케이블 : 실드 처리하고 덮음

#### 광섬유 케이블
- 광섬유로 만든 케이블
- 레이저를 이용해서 통신하기 때문에 구리선과는 비교할 수 없을 만큼의 장거리 및 고속 통신 가능
- 보통 100Gbps의 데이터 전송함
- 광섬유 내부와 외부를 다른 밀도를 가지는 유리나 플라스틱 섬유로 제작해서
- 한 번 들어간 빛이 내부에서 계속적으로 반사하며 전진하여 반대편 끝까지 가는 원리 이용
- 코어(core) : 빛의 굴절률이 높은 부분
- 클래딩(cladding) : 빛의 굴절률이 낮은 부분

### 무선 LAN(IEEE802.11)
- 수신과 송신에 같은 채널을 사용하기 때문에
- 반이중화 통신 사용

#### 반이중화 통신(half duplex)
- 양쪽 장치는 서로 통신할 수 있지만, 동시에는 통신할 수 없으며
- 한 번에 한 방향만 통신 가능한 방식
- 일반적으로 장치가 신호를 수신하기 시작하면 응답하기 전에 전송이 완료될 때까지 기다려야 함
- 또한, 둘 이상의 장치가 동시에 전송하면 충돌이 발생하여 메시지가 손실되거나 왜곡될 수 있기 때문에 충돌 방지 시스템 필요함

#### CSMA/CA
- 반이중화 통신 중 하나로 장치에서 데이터를 보내기 전에 일련의 과정을 기반으로 사전에 가능한 한 충돌을 방지하는 방식

### 무선 LAN(WLAN, Wireless Local Area Network) 이루는 주파수
- 무선 신호 전달 방식을 이용하여 2대 이상의 장치를 연결하는 기술
- 비유도 매체인 공기에 주파수를 쏘아 무선 통신망 구축
- 주파수 대역은 2.4GHz 대역 또는 5GHz 대역 중 하나를 써서 구축
- 2.4GHz : 장애물에 강한 특성을 가지고 있음, 전자레인지, 무선 등 전파 간섭이 일어나는 경우가 많음
- 5GHz : 사용할 수 있는 채널 수도 많고 동시에 사용 가능, 상대적으로 깨끗한 전파 환경 구축 가능
- 5GHz 대역을 사용하는 것이 좋음

#### 와이파이(wifi)
- 전자기기들이 무선 LAN 신호에 연결할 수 있게 하는 기술
- 이를 사용하려면 무선 접속 장치(AP, Access Point)가 있어야 함. 흔히 공유기라고 함
- 이를 통해 유선 LAN에 흐르는 신호를 무선 LAN 신호로 바꿔주어
- 신호가 닿는 범위 내에서 무선 인터넷을 사용할 수 있게 됨
- 참고 : 무선 LAN을 이용한 기술로는 와이파이만 있는 것이 아니고 지그비, 블루투스 등이 있음

#### BSS(Basic Service Set)
- 기본 서비스 집합
- 단순 공유기를 통해 네트워크에 접속하는 것이 아닌 
- 동일 BSS 내에 있는 AP들과 장치들이 서로 통신이 가능한 구조를 말함
- 근거리 무선 통신 제공
- 하나의 AP만을 기반으로 구축이 되어 있어 사용자가 한 곳에서 다른 곳으로 자유롭게 이동하며 네트워크에 접속하는 것을 불가능

#### ESS(Extended Service Set)
- 하나 이상의 연결된 BSS 그룹
- 장거리 무선 통신을 제공
- BSS보다 더 많은 가용성과 이동성 지원
- 사용자는 한 장소에서 다른 장소로 이동하며 중단 없이 네트워크에 계속 연결 가능

### 이더넷 프레임
- 데이터 링크 계층을 이더넷 프레임을 통해 전달받은 데이터의 에러를 검출하고 캡슐화함
- Preamble : 이더넷 프레임이 시작임을 알림
- SFD(Start Frame Delimiter) : 다음 바이트부터 MAC 주소 필드가 시작됨을 알림
- DMAC, SMAC : 수신, 송신 MAC 주소
- EtherType : 데이터 계층 위의 계층인 IP 프로토콜을 정의, ex)IPv4 or IPv6
- Payload : 전달받은 데이터
- CRC : 에러 확인 비트

✔️ MAC 주소 <br>
- 컴퓨터나 노트북 등 각 장치에는 네트워크에 연결하기 위한 장치(LAN 카드)가 있는데,
- 이를 구별하기 위한 식별변호
- 6바이트(48비트)로 구성됨

### 계층 간 데이터 송수신 과정
- 컴퓨터를 통해 다른 컴퓨터로 데이터를 요청한다면?
- ex) HTTP를 통해 웹 서버에 있는 데이터를 요청한다면?
- 애플리케이션 계층에서 전송 계층으로 요청(request) 값들이 캡슐화 과정을 거쳐 전달되고,
- 다시 링크 계층을 통해 해당 서버와 통신을 하고,
- 해당 서버의 링크 계층으로부터 애플리케이션까지 비캡슐화 과정을 커쳐 데이터가 전송됨

#### 캡슐화 과정
- 상위 계층의 헤더와 데이터를 하위 계층의 데이터 부분에 포함시키고
- 해당 계층의 헤더를 삽입하는 과정
- 애플리케이션 계층 -> 전송 계층 : '세그먼트' 또는 '데이터그램'화 되며 TCP(L4) 헤더가 붙여지게 됨
- -> 인터넷 계층 : IP(L3) 헤더가 붙여지게 되며 '패킷'화가 되고
- -> 링크 계층 : 프레임 헤더와 프레임 트레일러가 붙어 '프레임'화가 됨

#### 비캡슐화 과정
- 하위 계층에서 상위 계층으로 가며
- 각 계층의 헤더 부분을 제거하는 과정
- 캡슐화된 데이터를 받게 되면 링크 계층에서부터 타고 올라오면서
- 프레임화된 데이터는 다시 패킷화를 거쳐 세그먼트, 데이터그램화를 거쳐 메시지화가 되는 비캡슐화 과정 일어남
- 그 이후 최종적으로 사용자에게 애플리케이션의 PDU 메시지로 전달됨

---
# 2.2.2 PDU(Protocol Data Unit)
- 네트워크의 어떠한 계층에서 계층으로 데이터가 전달될 때 한 덩어리의 단위
- 헤더 : 제어 관련 정보들이 포함됨
- 페이로드 : 데이터를 의미

✔️ 계층마다 부르는 명칭이 다름 <br>
- 애플리케이션 계층 : 메시지
- 전송 계층 : 세그먼트(TCP), 데이터그램(UDP)
- 인터넷 계층 : 패킷
- 링크 계층 : 프레임(데이터 링크 계층), 비트(물리 계층)

✔️ 예시 <br>
- 애플리케이션 계층을 '메시지'를 기반으로 데이터를 전달하는데,
- HTTP의 헤더가 문자열이 것
- curl 명령어를 이용하여 www.naver.com으로 HTTP 요청을 해서 PDU 테스팅
- curl www.naver.com이란 명령어를 통해 요청
- 문자열 응답(response) 헤더 값이 나옴
- PDU 중 아래 계층인 비트로 송수신 하는 것이 모든 PDU 중 가장 빠르고 효율성이 높음.
- 하지만 애플리케이션 계층에서는 문자열을 기반으로 송수신을 하는데, 
- 그 이유는 헤더에 authorizaion 값 등 다른 값들을 넣는 확장이 쉽기 때문

## ✅ TCP/IP 4계층 면접 질문 & 답변

### 🔹 Q1. TCP/IP 4계층과 OSI 7계층의 차이점은 무엇인가요?

**A.**
TCP/IP 모델은 실질적인 구현을 중심으로 4계층으로 구성되며,
OSI 모델은 이론적이고 교육적인 목적으로 7계층으로 더 세분화되어 있습니다.

* TCP/IP의 **애플리케이션 계층**은 OSI의 \*\*상위 3계층(응용, 표현, 세션)\*\*을 포함합니다.
* TCP/IP의 **링크 계층**은 OSI의 **데이터 링크 + 물리 계층**을 합친 개념입니다.

### 🔹 Q2. TCP와 UDP의 차이점을 설명해주세요.

**A.**

| 항목    | TCP                        | UDP                 |
| ----- | -------------------------- | ------------------- |
| 연결 방식 | 연결 지향 (3-way handshake)    | 비연결 지향              |
| 신뢰성   | 수신 확인(ACK), 순서 보장          | 보장 없음               |
| 속도    | 느림 (오버헤드 존재)               | 빠름 (간단한 구조)         |
| 사용 예  | 웹, 파일전송(HTTP, SMTP, SSH 등) | 실시간 스트리밍, DNS, 게임 등 |

> TCP는 신뢰성이 있고, UDP는 속도가 중요할 때 사용합니다.

### 🔹 Q3. TCP의 3-way-handshake는 왜 필요한가요?

**A.**
신뢰성 있는 통신을 위해 클라이언트와 서버가 **연결을 협의하고, 초기 시퀀스 번호(ISN)를 주고받기 위한 과정**입니다.
이를 통해 **패킷 순서, 데이터 손실 방지, 중복 연결 방지**가 가능합니다.

### 🔹 Q4. TCP의 4-way-handshake에서 TIME\_WAIT 상태는 왜 존재하나요?

**A.**

* **지연된 패킷이 다시 들어올 경우 대비**
* **상대방의 FIN이 잘 도착했는지 확인할 시간 확보**

즉, 데이터 무결성과 연결 안정성을 위해 일정 시간 동안 소켓을 유지합니다.
운영체제마다 시간은 다르며 (보통 1\~4분), 이를 통해 예기치 않은 충돌을 방지합니다.

### 🔹 Q5. 전송 계층에서 TCP와 UDP 외에 다른 프로토콜도 있나요?

**A.**
네, 대표적으로 **QUIC**(구글 개발)가 있습니다.
UDP 기반이지만 **TLS + 0-RTT 핸드셰이크**로 빠르고 보안도 보장되며, 최근 HTTP/3에서 사용됩니다.

### 🔹 Q6. 데이터가 애플리케이션 계층에서 링크 계층까지 전달될 때 어떤 과정을 거치나요?

**A.**
이 과정을 **캡슐화(encapsulation)** 라고 합니다.
각 계층마다 **헤더가 붙으며** 형태가 바뀝니다.

```
애플리케이션 계층 → 메시지  
전송 계층 → 세그먼트(TCP)  
인터넷 계층 → 패킷  
링크 계층 → 프레임
```

반대로 수신 쪽에서는 **비캡슐화** 과정이 일어나며, 헤더를 하나씩 제거하고 원본 데이터만 꺼냅니다.

### 🔹 Q7. 이더넷 프레임이란 무엇인가요?

**A.**
링크 계층에서 데이터를 전달할 때 사용하는 기본 단위입니다.
프레임은 **MAC 주소, 데이터, 에러 검출 코드(CRC)** 등을 포함합니다.

* Preamble, SFD → 시작 알림
* DMAC/SMAC → 수신/송신 MAC 주소
* EtherType → 상위 계층 지정(IP 등)

### 🔹 Q8. 전이중(Full Duplex)과 반이중(Half Duplex)의 차이점은?

**A.**

* **전이중**: 양방향 동시에 통신 가능 (예: 유선 LAN)
* **반이중**: 한 번에 한 방향만 통신 가능 (예: 무선 LAN, 무전기)

전이중은 충돌이 없지만, 반이중은 충돌이 발생할 수 있어 **CSMA/CA** 같은 충돌 방지 방식이 필요합니다.

### 🔹 Q9. CSMA/CD와 CSMA/CA의 차이는?

| 항목    | CSMA/CD               | CSMA/CA             |
| ----- | --------------------- | ------------------- |
| 방식    | 충돌 후 감지               | 충돌 전 예방             |
| 사용 환경 | 유선 LAN (이더넷, 예전 허브 등) | 무선 LAN (Wi-Fi, 공유기) |
| 특징    | 충돌 발생 후 재전송           | 전송 전에 기다림, 백오프 사용   |

### 🔹 Q10. 무선 LAN에서 2.4GHz와 5GHz 대역의 차이는?

| 항목     | 2.4GHz                 | 5GHz        |
| ------ | ---------------------- | ----------- |
| 속도     | 낮음                     | 높음          |
| 거리/장애물 | 멀리 감, 장애물에 강함          | 가까운 거리에서 빠름 |
| 간섭     | 높음 (전자레인지, 블루투스 등과 충돌) | 낮음          |

### 🔹 Q11. TCP/IP 모델에서 IP는 신뢰성 있는가요?

**A.**
아니요.
**IP는 비연결형이며, 수신 확인이나 재전송을 하지 않습니다.**
즉, **빠르지만 신뢰성이 없습니다.**
그래서 IP 위에 TCP 같은 **신뢰성 있는 전송 계층**이 필요합니다.

### 🔹 Q12. MAC 주소는 왜 필요한가요?

**A.**
MAC 주소는 **네트워크 인터페이스 카드(NIC)에 부여된 고유 식별자**입니다.

* LAN 내부에서 **장치를 구분**하고
* **이더넷 프레임의 출발지/도착지 주소**로 사용됩니다.
  IP가 논리적 주소라면, MAC은 **물리적 주소**입니다.

### 🔹 Q13. 무선 LAN에서 ESS와 BSS의 차이는?

| 항목  | BSS (Basic Service Set) | ESS (Extended Service Set) |
| --- | ----------------------- | -------------------------- |
| 구성  | 1개의 AP                  | 여러 개의 AP로 구성된 네트워크 그룹      |
| 이동성 | 없음                      | 가능 (로밍, 끊김 없는 연결)          |
| 적용  | 소형 환경 (집, 소규모 매장)       | 대형 환경 (학교, 회사, 공공기관)       |

필요하시면 이 내용을 PDF나 슬라이드 포맷, 또는 토픽별 스터디 자료로도 정리해드릴게요!
또한 퀴즈나 요약 카드 형식도 제작 가능합니다. 원하시나요?

## ✅ TCP/IP 4계층 관련 심화 문제 (면접/스터디용)

### 🔸 Q1. TCP 통신에서 서버가 FIN을 보냈지만, 클라이언트는 ACK 응답을 보내지 않았습니다.

📌 **핵심 개념**: 4-way handshake, LAST\_ACK 상태
✅ **정답**:
서버는 `LAST_ACK` 상태에 머무르며 클라이언트의 ACK를 기다립니다. 이는 신뢰성 있는 연결 종료를 보장하기 위한 설계로, 클라이언트의 응답이 누락되었을 경우 재전송을 통해 데이터 유실을 방지합니다.

### 🔸 Q2. DNS 요청 실패 후 TCP 연결도 실패한 경우, 각각 어느 계층 문제인가요?

📌 **핵심 개념**: 애플리케이션 계층(DNS), 전송 계층(TCP)
✅ **정답**:
DNS는 **애플리케이션 계층**, TCP 연결은 **전송 계층**의 문제입니다. DNS가 실패하면 IP 주소를 얻지 못해 TCP 연결 자체가 불가능해집니다.

### 🔸 Q3. HTTP 요청 패킷 일부만 수신된 경우, TCP는 어떻게 보완하나요?

📌 **핵심 개념**: 세그먼트, 순서 보장, 재전송
✅ **정답**:
TCP는 시퀀스 번호로 패킷의 순서를 맞추고, 누락된 세그먼트는 자동으로 재전송합니다. 이를 통해 전체 데이터가 올바르게 재구성됩니다.

### 🔸 Q4. UDP가 실시간 스트리밍에 적합한 이유는?

📌 **핵심 개념**: 지연 최소화, 오버헤드 없음
✅ **정답**:
UDP는 핸드셰이크 없이 바로 데이터를 전송하며 재전송하지 않기 때문에 **지연이 적고 빠릅니다**. 일부 손실은 감수하더라도 빠른 응답이 중요한 스트리밍에 유리합니다.

### 🔸 Q5. Wi-Fi 느림 문제 진단 시 ping, netstat, traceroute로 무엇을 확인할 수 있나요?

📌 **핵심 개념**: 계층별 도구 기능
✅ **정답**:

* `ping`: 인터넷 계층, 응답 시간/손실 확인
* `netstat`: 전송 계층, 포트 상태/세션 확인
* `traceroute`: 인터넷 계층, 병목 구간 확인

### 🔸 Q6. IP 스푸핑이 UDP에서 더 많이 활용되는 이유는?

📌 **핵심 개념**: 비연결형, 인증 없음
✅ **정답**:
UDP는 수신 확인 없이 바로 전송되므로 **발신 IP 위조**가 탐지되지 않고도 공격이 가능합니다. 반면 TCP는 3-way handshake로 응답을 받아야 하므로 스푸핑이 어렵습니다.

### 🔸 Q7. 무선 LAN에서 CSMA/CA를 사용하는 이유는?

📌 **핵심 개념**: 무선 충돌 방지, 반이중 통신
✅ **정답**:
무선 환경에서는 충돌 감지가 어려워 **충돌을 사전에 방지하는 CSMA/CA**를 사용합니다. 전이중 통신(유선)은 송수신 경로가 분리되어 충돌 자체가 없으므로 해당 메커니즘이 필요 없습니다.

### 🔸 Q8. 흐름 제어와 혼잡 제어의 차이는?

📌 **핵심 개념**: 수신자 vs 네트워크 혼잡
✅ **정답**:

* **흐름 제어**: 수신자가 처리할 수 있는 속도에 맞춰 전송 제한
* **혼잡 제어**: 네트워크 내 과부하를 방지해 패킷 손실 줄임
  → TCP는 수신 윈도우와 혼잡 윈도우를 함께 사용하여 전송 속도를 조절합니다.

### 🔸 Q9. IP 패킷 유실 시 복구는 어떻게 이루어지나요?

📌 **핵심 개념**: 비신뢰형 IP, TCP 재전송
✅ **정답**:
IP 계층은 유실된 패킷을 복구하지 않습니다. 상위 전송 계층의 TCP가 **시퀀스 번호로 누락을 감지**하고 재전송을 통해 복구합니다.

### 🔸 Q10. HTTP/3에서 QUIC(UDP 기반)를 사용해도 신뢰성을 확보할 수 있는 이유는?

📌 **핵심 개념**: QUIC, TLS 내장, 멀티스트림
✅ **정답**:
QUIC은 UDP 위에 자체 재전송, 순서 보장, 암호화(TLS 1.3)를 직접 구현합니다. 0-RTT 연결로 빠르게 통신을 시작하면서도 **TCP 수준의 신뢰성**과 보안을 제공합니다.
